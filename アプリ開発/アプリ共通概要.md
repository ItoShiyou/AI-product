# 16アプリ共通開発・基盤定義書 (Engineering Specification)

## 1. 開発の全体目的

ミニマリスト的思考に基づき、**「最小限の機能、最大限の統一感」**を持つ16個のツールアプリを迅速に展開する。

* **一貫したユーザー体験**: 全アプリで共通の操作感（デザイン・インタラクション）を提供。
* **開発効率の極大化**: デザインシステムとネイティブ・ブリッジを共通化し、1つのアプリを量産する速度で16個を構築する。
* **保守性の確保**: 1つの共通コード（CSS/JSモジュール）の修正が、全てのアプリに波及する設計。

---

## 2. 技術スタック (Technical Stack)

* **Frontend**: HTML5 / CSS3 / JavaScript (ES6+)
* **Framework**: Flutter または React Native (RN) による **WebViewラップ + Native Bridge**
* *注: 全OSで同一の見た目を実現するため、UIはすべてWeb側で描画する。*


* **Storage**: 各OSのLocal File System または SQLite (Bridgeを介してアクセス)

---

## 3. 共通化・カプセル化の方針

### A. 統合CSSシステム (Themed CSS Variables)

全てのアプリは単一の `core.css` を参照する。デザインの変更は変数（Variables）の書き換えのみで行う。

* **CSS変数による管理**:
* `--primary-color`: 各アプリ固有のメインカラー
* `--bg-color`: ライト/ダークモード別の背景色
* `--font-main`: 文字色（コントラスト比 4.5:1 以上を保持）
* `--spacing-unit`: 8px単位のグリッドシステム


* **UIコンポーネントの標準化**:
* `button.btn-primary`, `input.form-control`, `header.app-bar` 等のクラスを全アプリで共通化。



### B. ネイティブ・ブリッジ・モジュール (Universal Bridge)

JSからネイティブ機能を呼び出すためのインターフェースを抽象化する。

```javascript
// 共通インターフェースのイメージ
const NativeTool = {
  saveData: (key, data) => { /* Bridge logic */ },
  openCamera: () => { /* Bridge logic (QR/ColorPicker用) */ },
  notify: (title, body) => { /* Timer通知用 */ }
};

```

### C. 共通ユーティリティライブラリ

* **計算・変換ロジック**: 丸め処理、単位変換、税計算などの共通関数。
* **時刻管理**: タイマー、ストップウォッチ、カレンダー計算の共通ロジック。

---

## 4. プロジェクト構造（Monorepo構成案）

16個のアプリを効率的に管理するため、モノリポ構成を推奨する。

```text
/root
  ├── /shared
  │     ├── /css        # core.css, theme.css
  │     ├── /js         # storage.js, bridge.js, utils.js
  │     └── /assets     # 共通アイコンセット, フォント
  ├── /apps
  │     ├── /01_memo
  │     ├── /02_todo
  │     └── ... (16 apps)
  └── /native_wrapper   # Flutter/RNのソースコード

```

---

## 5. エンジニアへの重要指示事項

1. **OS標準UIの無効化**:
* フォーム要素（Input, Select等）のデフォルトスタイルを完全にリセット（`appearance: none`）し、共通CSSで再定義すること。
* スクロール挙動やバウンス効果をWeb側で制御し、OS間の差異を最小化すること。


2. **アクセシビリティの数値管理**:
* パステルカラーの背景に対して、動的に文字色のコントラスト比を計算、あるいは定義された文字色を強制適用する仕組みを実装すること。


3. **ダークモードの動的切替**:
* `prefers-color-scheme` への追従だけでなく、アプリ内の設定から各モード専用パレットへ瞬時に切り替え可能にすること。


4. **アセットの軽量化**:
* 16個のアプリを統合する際、リソースを圧迫しないようアイコンはすべてSVGコードとして管理すること。

# 追加差分
## 1. bondaviデザインの核心を解剖する

彼らのアプリが「なぜ心地よいのか」を論理的に分析すると、以下の3点に集約されます。

* **「白」ではなく「余白」をデザインしている**:
画面の50%以上が何も書かれていない空間です。要素を中央に寄せ、情報の密度を極限まで下げています。
* **「機能」を「一言」に集約している**:
例えば「設定」ではなく「いじる」といった、親しみやすい言葉選びと、1画面1アクションの原則。
* **不規則なパステルカラーと手書き感**:
幾何学的な完璧さよりも、少し角を丸め、目に優しい（彩度を抑えた）配色が特徴です。

---

## 2. 実装に落とし込むための3つのステップ

エンジニアに渡すための、bondavi風デザインの実装ガイドラインです。

### ① 「角丸」と「ボーダー」の禁止

bondaviデザインの多くは、枠線（Border）を使いません。

* **指示**: 「要素の区切りに線を使うのをやめ、**色の面（Surface）の切り替え**、または**大胆な余白**だけで構成してください。」
* **CSS値**: `border: none;` / `border-radius: 20px;`（かなり大きめの角丸）

### ② フォントと「文字間隔」の調整

文字が詰まっているとミニマリズムは崩壊します。

* **指示**: 「日本語フォントの `letter-spacing`（字間）を0.05em〜0.1em程度広げ、`line-height`（行間）を1.8倍以上に設定してください。これにより『抜け感』が生まれます。」

### ③ ユーザーを迷わせない「一点集中UI」

* **指示**: 「1画面に配置するボタンは、原則として**1つ（メインアクションのみ）**。それ以外のサブ機能は、目立たないテキストリンクにするか、設定画面へ隠してください。」

---

## 3. bondavi風パステル・カラーパレットの構築

彼らの配色の特徴は、**「低彩度・高明度」**かつ**「少しグレーが混じったようなくすみ」**です。

先ほど作成した16色のリストを、以下のルールで調整するようデザイナーに伝えてください。

* **ベース彩度の制限**: HSL形式で、S（彩度）を30%〜50%以内に抑える。
* **輝度の固定**: L（輝度）を85%〜95%の範囲で固定する。
これで、16色あっても全てが同じトーン（同じ明るさのトーン）に見えるようになります。

---

## 4. 模倣から「一貫性」への昇華（エンジニアへの指示）

bondaviの模倣を16個のアプリで成功させるための、RN/Webブリッジの最終指示です。

> **「UIコンポーネントの完全固定」**
> 16個すべてのアプリで、以下のCSSクラスを1pxの狂いもなく共有してください。
> * `.ba-card`: 背景パステル、角丸20px、影なし。
> * `.ba-title`: フォントサイズ24px、太字、中央揃え、上下に大きなマージン。
> * `.ba-button`: 各アプリのメインカラー、文字は白（またはダークグレー）、タップ時にわずかに縮小するアニメーション。
> 
>

### 論理的な指摘

bondaviのデザインは「引き算」が完璧であるため、少しでも要素（広告、複雑な設定、多すぎるボタン）が増えると、途端に「安っぽいアプリ」に見えてしまうリスクがあります。16個のアプリを作る際、**「機能を削る勇気」**が最も重要な開発要件となります。
